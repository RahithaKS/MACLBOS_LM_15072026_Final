What's happening

Role	What it allows

Now with the full picture — code downloaded from Replit → run locally → Docker image → pushed to ACR → Azure App Service pulls it, with Azure PostgreSQL Serverless + Key Vault — here are all possible issues categorized:

1. Session Cookie Broken (Most Likely for the SSO Bug)
The problem: Azure App Service sits behind Microsoft's own reverse proxy/load balancer. Traffic from the browser arrives as HTTPS, but inside the container the app sees it as HTTP.

Your session config in server/index.ts:

cookie: {
  secure: process.env.NODE_ENV === 'production',  // ← true in production
  httpOnly: true,
}

With secure: true, Express will only send the session cookie over HTTPS. But since the container sees HTTP internally, Express thinks it's an HTTP connection → may refuse to set the cookie OR the cookie gets dropped.

Fix needed in server/index.ts:

app.set('trust proxy', 1);  // ← add this BEFORE session middleware

This tells Express to trust Azure's x-forwarded-proto: https header, so it correctly knows the connection IS HTTPS from the browser's perspective.

Without this, req.session.regenerate() → res.redirect('/dashboard') sets a session cookie but the browser never stores it because the Secure flag gets confused. This is why standard users lose their session immediately.

2. SameSite Cookie Issue with SSO Redirect Chain
The problem: Microsoft SSO is a cross-site redirect. The flow is:

Your app → login.microsoftonline.com → Your app (callback)

The final redirect from Microsoft back to your callback is a cross-site top-level navigation. Modern browsers default to SameSite=Lax. This should be fine for GET-based redirects — but if SameSite isn't explicitly set, some browsers (especially in corporate environments like Bosch's managed Chrome/Edge) may behave differently.

Fix: Explicitly set SameSite:

cookie: {
  secure: true,
  httpOnly: true,
  sameSite: 'lax',   // ← add this explicitly
  maxAge: 30 * 24 * 60 * 60 * 1000,
}

3. PostgreSQL Serverless — Cold Start Kills First Requests
The problem: Azure PostgreSQL Serverless pauses after inactivity (typically 5–10 minutes). When a user logs in after the DB has been idle, the first few DB queries (session lookup, user lookup) hit a cold start that can take 15–30 seconds.

What you'll see:

SSO callback → exchangeCodeForUser works (calls Microsoft, no DB) → then storage.getDomainUserByEmail() hangs → request times out → res.redirect('/?sso_error=server_error')
Or: session write (req.session.regenerate()) times out → empty session → 401 on next request
Fix options:

Set a connection pool timeout and retry in db.ts
Increase App Service's health check timeout
Switch to General Purpose (non-serverless) tier to eliminate cold starts
4. Key Vault Secret Loading — Race Condition at Startup
The problem: Your container starts, Express begins accepting requests immediately. But if Key Vault secrets (like SESSION_SECRET, DATABASE_URL, OpenAI keys) are loaded asynchronously at startup, there's a window where:

App is "running" from Azure's health check perspective
But SESSION_SECRET hasn't loaded yet
First users to hit the app get broken sessions (random secret was used, then the real one loaded)
Fix: Make sure Key Vault secrets are loaded before app.listen() is called. The startup sequence must be:

1. Load Key Vault secrets
2. Initialize DB pool
3. Run migrations
4. app.listen()

If step 4 happens before step 1 completes, first requests are broken.

5. PostgreSQL SSL Connection String
The problem: Azure PostgreSQL requires SSL. Your server/index.ts session pool config:

const isLocalDb = process.env.DATABASE_URL?.includes('localhost') || 
                  process.env.DATABASE_URL?.includes('ledgerlm-db') || ...

This should be fine for Azure PostgreSQL — rejectUnauthorized: false allows self-signed certs. But the session pool and the main app DB pool (db.ts) must BOTH use the same SSL setting. If only one is configured correctly, you'll get intermittent connection errors.

Also: Azure PostgreSQL Serverless has a connection limit (usually 50–100 for small SKUs). Your app has:

1 session pool (for connect-pg-simple)
1 main Drizzle pool
Each pool creates multiple connections. If the Python backend also connects to Postgres, you can hit the limit under load.

6. Python Backend — Separate Container or Same Container?
The problem: Your app has a server/python-backend.ts that starts a Python FastAPI/Flask process. When you run locally, it starts as a child process on the same machine. In a Docker container pushed to ACR:

If it's a single container, the Python backend must be started correctly by the Node.js process as a subprocess. The container entrypoint must allow this.
If it's a separate container, you need an internal network URL (PYTHON_BACKEND_URL=http://localhost:5001 won't work — needs the actual container name or Azure internal hostname).

8. Container Restart Wipes In-Memory State

MSAL (@azure/msal-node) caches tokens in memory by default. After a restart, the token cache is empty. This is fine for user-facing flows (they just re-authenticate) but can cause issues if your backend uses client-credentials tokens for things like Azure Blob access or Graph API calls, which would need to re-acquire tokens after every restart.
